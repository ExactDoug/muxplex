## Feature 1 — New session auto-becomes the active session

**Goal:** after creating a session from the UI, the UI should switch to it (fullscreen
terminal), not leave the user on the grid.

**Current state (partially there — AUDIT why it doesn't always stick):**
- `createNewSession(name, remoteId, viewNames)` — `app.js:5425`. It shows a loading tile
  (`app.js:5476`), then a `pollForSession` `setInterval` (`app.js:5503`) that waits up to
  ~30s for the session to appear in `_currentSessions`, and on success **already calls
  `openSession(sessionName, { remoteId: deviceId })`** (`app.js:5513`) before
  `removeLoadingTile()`. So the happy path is implemented.
- **Therefore the task is mostly an audit + fix of the gaps**, not a greenfield build.
  Investigate:
  1. **Other creation entry points** that may NOT route through `createNewSession`'s poll
     loop: `showNewSessionInput` / `showFabSessionInput` (inline name inputs), the
     expanded-header new-session button (v0.7.0), and the FAB. Confirm each ends in the
     same `openSession`-on-success path.
  2. **Timeout path** (`app.js:5515`+) — if the session is slow, the interval clears
     without opening. Decide whether to keep waiting / show a retry.
  3. **Poll-race**: `openSession` is called from inside `setInterval` after a
     `pollSessions()`; verify `_currentSessions` actually contains the new session's
     `sessionKey` (`expectedKey` at `app.js:5440`) so the right one opens.
  4. Whether a subsequent poll/`renderGrid` or view-filter switch *resets* the view back
     to grid right after opening.

**Acceptance:** creating a session from *every* UI entry point lands the user in that
session's fullscreen terminal, for both local and remote/federated targets.

## Feature 2 — Rename tmux sessions from the UI

**Goal:** rename a live tmux session, reachable from (a) the session pills in **both**
header variants (main-page header + expanded terminal header) and (b) the tile/card
(grid + sidebar) flyout menu.

**Current state:** **no session rename exists** (frontend or backend). Only *View* rename
exists — reuse it as the UX template.

### Backend (does not exist — build it)
- No `rename` endpoint in `main.py`; session endpoints live around `app.js`… → see
  `main.py:604` (`GET /api/sessions`), `main.py:637` (`POST /api/sessions`
  `create_session`). Add e.g. `POST /api/sessions/{name}/rename` (+ the federation proxy
  variant mirroring the `/api/federation/{device}/connect/{name}` pattern in
  `app.js:3478`-style routing).
- tmux mechanism: `tmux rename-session -t <old> <new>` via the `run_tmux` helper
  (imported in `main.py:58`; tmux command layer is `muxplex/ttyd.py`). ttyd attaches by
  session (`tmux attach -t <name>`, `ttyd.py:176`) — an existing attach survives a rename,
  but a *re-spawn* needs the new name; verify the active ttyd/terminal after rename.
- **Validation:** reject empty/whitespace, duplicates, and the reserved **`dir:`** prefix
  (auto-view namespace — see CLAUDE.md frontend contract §6) and any tmux-illegal chars
  (`.`, `:` are special in tmux targets).

### ⚠️ Critical cascade (the main design risk)
A session is identified by `sessionKey = device_id:name` (`main.py` ~297/628). **Views and
hidden-state store membership by key/bare name** (`views.py:71`, and `hidden_sessions`).
Renaming changes the key, so the rename MUST atomically update:
- every `view.sessions` entry that referenced the old key/name (`views.py`,
  `settings.py`),
- `hidden_sessions` entries,
- the persisted active-session state (`active_session` / `active_remote_id`, see
  `app.js:3432` PATCH `/api/state`) if the renamed session is current,
- federation sync implications (renames must propagate / not desync remote peers).

Decide: is rename **local-only** (disable for remote sessions in v0.9) or federated? Local
-only is a reasonable v0.9 scope cut — state it explicitly if chosen.

### Frontend UX (mirror existing inline-rename)
- **Template to copy:** View inline rename in `renderManageViewList` /
  `app.js:2836`-`2905` (`commitRename`/`revertRename`, Enter commits, Escape reverts,
  blur commits). Also the inline session-name input (`showNewSessionInput`).
- **Tile/card:** add a `Rename…` action to the flyout menu items at `app.js:152`-`167`
  (currently `Add to View…`, `Kill Session`); dispatch in the flyout `data-action` handler
  (`app.js:211`).
- **Pills:** add `Rename…` to the pill dropdown menus — the view-dropdown
  (`renderViewDropdown`) and the expanded-header pill menu (`_epMenuItemHTML`,
  ~`app.js:3760`; opened via `_epToggleMenu`). The current pill is a no-op on click
  (CLAUDE.md contract), so rename belongs in its **dropdown**, not the pill click.
- Respect contract §3 (no listener stacking — container listeners attach once).

**Acceptance:** rename a session from a tile flyout AND from both header pill dropdowns;
the new name shows everywhere (grid, sidebar, pills, title); view membership, hidden
state, and the active session survive the rename with no orphaning.

## Feature 3 — Visually distinguish View pills from session pills in the headers

**Goal:** in the top headers a *View* pill is currently too similar to a *session* pill —
make views obviously a different kind of thing at a glance.

**Current state:**
- **Main-page header** view pills: `renderViewPills()` `app.js:1215`; auto-view pills
  `_autoViewPillHTML` `app.js:1277` already use `nav-pill--auto` (dashed + 📁, CHANGELOG
  v0.8.0). User-view pills currently lean on the base `.nav-pill` look.
- **Expanded header:** session pills `_epSessionPillHTML` → `nav-pill nav-pill--session`
  (+`--current`) at `app.js:3789`; view-group pills via `_epMenuPillHTML('view:'+i, …)`
  with `--auto` only for auto-views (`app.js:3886`, `3902`).
- **CSS:** `style.css` — `.nav-pill` (2196), `--current` (2223), `--other` (2243),
  `--auto` (2249). There is **no `.nav-pill--view`** treatment for user views distinct
  from sessions.

**Direction (design in the next chat):** introduce a clear visual language —
e.g. a `nav-pill--view` modifier with a distinct shape/affordance (leading layers/stack or
folder-style glyph, different border-radius or accent, or a subtle container) so View vs
Session reads instantly, while staying consistent with the existing `--auto` (dashed/📁)
and `--current` treatments. Apply in **both** header variants. Keep it accessible
(don't rely on color alone — add an icon/shape cue). Mind the width-aware pill collapse
(contract §5/§6) so new affordances don't break the fit algorithm
(`allocateExpandedPills`, `_epMeasureWidth`).

**Acceptance:** at a glance, View pills are unmistakably different from session pills in
both headers; auto-views remain distinct from user-views; collapse/overflow still works.
